Setup Guide — Cloud & DevOps Engineering Part II

End-to-end instructions: from an empty Azure subscription to a live, pipeline-deployed Flask application. Written so that someone testing this project from scratch (e.g. for grading) can follow it without needing prior context from development sessions.

1. Choose a unique project name — and write it down

Storage Account and Key Vault names must be globally unique across all of Azure and have strict length limits. Pick one short, lowercase, alphanumeric name and use it everywhere — local terraform.tfvars, Azure DevOps pipeline variables, all of it. Mixing different values between local and pipeline runs is the single most common source of errors in this project (Terraform will try to destroy and recreate everything under the new name, since Azure resource names cannot be renamed in place).

$PROJECT = "yourname42"   # example - replace with your own, keep it short

Length budget: the tightest constraint is the Key Vault name (kv-<project_name>-<environment>, max 24 characters). With environment = "dev" (3 chars) and the kv-- separators (4 chars), that leaves at most 17 characters for project_name.

Use a different, separate name for the remote-state bootstrap (it has its own, independent uniqueness requirement):

$BOOTSTRAP_PROJECT = "yourname42state"

2. Bootstrap the remote state backend (one-time)

cd bootstrap
terraform init
terraform apply -var="project_name=$BOOTSTRAP_PROJECT"

Confirm with yes. Note the outputs:

resource_group_name  = "rg-<bootstrap_project>-tfstate"
storage_account_name = "sttfstate<bootstrap_project>"

Write these down — they are needed in step 3 and step 7.

cd ..

3. Configure and apply the main infrastructure locally

Create terraform.tfvars in the project root:

project_name = "yourname42"
environment  = "dev"
location     = "swedencentral"

Initialize with the backend from step 2:

terraform init `
  -backend-config="resource_group_name=rg-yourname42state-tfstate" `
  -backend-config="storage_account_name=sttfstateyourname42state" `
  -backend-config="container_name=tfstate" `
  -backend-config="key=cloud-project.tfstate"

terraform plan
terraform apply

Confirm with yes. This creates: Resource Group, Storage Account + images container, Key Vault (RBAC) + secret, App Service Plan, Linux Web App with a System-Assigned Managed Identity, and the two RBAC role assignments that give that identity access to Storage and Key Vault.

Note the outputs — you’ll need app_service_name and resource_group_name later.

4. Grant yourself data-plane access (for local app testing only)

Terraform’s RBAC role assignments target the App Service’s Managed Identity, not your own az login account. To run the Flask app locally against real Azure Storage, grant yourself the same role:

$myObjectId = az ad signed-in-user show --query id -o tsv

az role assignment create `
  --assignee $myObjectId `
  --role "Storage Blob Data Contributor" `
  --scope "/subscriptions/<subscription-id>/resourceGroups/rg-yourname42-dev/providers/Microsoft.Storage/storageAccounts/styourname42dev"

Wait ~5 minutes for RBAC propagation before testing.

6. Deploy the app manually (fastest path to a working URL)

.\scripts\deploy.ps1 -ResourceGroupName "rg-yourname42-dev" -AppServiceName "app-yourname42-dev"

This already gives you a fully working application — steps 7–10 below are only needed to also demonstrate the CI/CD pipeline.

7. Set up the Azure DevOps Service Connection (one-time)

  1. Azure DevOps → Project Settings → Service connections → New service connection
  2. Azure Resource Manager → App registration (automatic) → Workload identity federation
  3. Scope level: Subscription, select your subscription
  4. Name: cloud-project-connection
  5. Save

This creates a Microsoft Entra App Registration + Service Principal with a federated credential trust — no client secret is ever generated.

Find the Service Principal’s Object ID: Service connection → Manage App registration → Overview → copy Object ID (not the Application/Client ID).

8. Grant the pipeline’s Service Principal the roles it needs

The pipeline needs more than Contributor, because Terraform also creates/deletes RBAC role assignments and reads/writes Key Vault secrets — neither of which Contributor covers (RBAC role management and Key Vault data-plane access are deliberately separated from general resource management in Azure).

$spObjectId = "<object-id-from-step-7>"
$subId = az account show --query id -o tsv

# Manage resources
az role assignment create --assignee $spObjectId --role "Contributor" --scope "/subscriptions/$subId"

# Create/delete RBAC role assignments (needed because Terraform manages
# the App Service's access to Storage/Key Vault)
az role assignment create --assignee $spObjectId --role "User Access Administrator" --scope "/subscriptions/$subId"

After the Key Vault exists (post first apply), also grant data-plane access to read/write the secret Terraform manages there:

az role assignment create `
  --assignee $spObjectId `
  --role "Key Vault Secrets Officer" `
  --scope "/subscriptions/$subId/resourceGroups/rg-yourname42-dev/providers/Microsoft.KeyVault/vaults/kv-yourname42-dev"

Wait ~5 minutes for propagation before running the pipeline.

9. Install the Terraform extension and set pipeline variables

Install the Marketplace extension (one-time, needs org admin rights): https://marketplace.visualstudio.com/items?itemName=ms-devlabs.custom-terraform-tasks → “Get it free” → select your organization → “Install”

Set pipeline variables (Pipelines → Edit → Variables):

Variable Value
PROJECT_NAME yourname42
TF_STATE_RESOURCE_GROUP rg-yourname42state-tfstate
TF_STATE_STORAGE_ACCOUNT sttfstateyourname42state
RESOURCE_GROUP_NAME rg-yourname42-dev
APP_SERVICE_NAME app-yourname42-dev

PROJECT_NAME is critical and easy to miss: terraform.tfvars is git-ignored, so the pipeline never sees it. Without explicitly passing project_name via this pipeline variable, Terraform silently falls back to the default value in variables.tf, which then drifts from whatever you’re using locally — causing Terraform to plan a full destroy-and-recreate under the “other” name on every alternating local/pipeline run.

10. Run the pipeline and verify

Create the pipeline (Pipelines → New Pipeline → select your repo → “Existing Azure Pipelines YAML file” → pipelines/azure-pipelines.yml), then run it.

Three stages run in order: 1. Terraform – Validate & Plan 2. Terraform – Apply (only on main) 3. Build & Deploy Application

If you see terraform init “stuck” with no error

Confirm -input=false is present on all terraform init calls (prevents hanging on an interactive prompt with no TTY available).

If TerraformApply fails with “initialized in an empty directory”

deployment jobs in Azure DevOps do not auto-checkout the repo like plain job blocks do. Confirm - checkout: self is the first step in both deployment jobs.

If you see RoleAssignmentsClient#Delete: 403

The Service Principal lacks User Access Administrator — see step 8.

If you see GetSecret: 403 ForbiddenByRbac

The Service Principal lacks Key Vault Secrets Officer on the Key Vault — see step 8 (must be granted after the vault exists).

If Deploy to App Service fails with “Resource group … could not be found (404)”

This is the task’s (slightly misleading) error for “I can’t see this resource at all” — almost always means RESOURCE_GROUP_NAME / APP_SERVICE_NAME pipeline variables don’t match what Terraform actually created (see the PROJECT_NAME consistency requirement in step 9), not a Basic Auth/publishing-credentials issue, even though some Stack Overflow threads suggest that. Verify with:

az group show --name <RESOURCE_GROUP_NAME-value>

If that itself returns ResourceGroupNotFound, the names are out of sync — fix PROJECT_NAME and re-run Terraform before touching the deploy step.

11. Verify the live application

Check Expected
https://app-yourname42-dev.azurewebsites.net/ “Files in storage” page loads
.../upload Upload form loads
Upload a test file Appears back on Page 1 with a download link
.../healthz {"status": "ok"}
Azure Portal → Key Vault → Access control (IAM) Shows RBAC role assignments, not Access Policies
Azure Portal → App Service → Identity “System assigned” → On

First request after a fresh deployment can take 1–3 minutes (cold start: Oryx build artifacts settle, Gunicorn workers start, the Managed Identity token handshake with IMDS completes). This is expected — wait and reload rather than repeatedly force-reloading, which starts multiple parallel cold starts. If it’s still not responding after ~5 minutes:

az webapp log tail --name app-yourname42-dev --resource-group rg-yourname42-dev

Key lesson for repeat runs

Pick one project_name, write it down once, and never deviate — in terraform.tfvars locally, in the PROJECT_NAME pipeline variable, and in every RBAC scope you assign. The overwhelming majority of issues encountered in this project’s setup traced back to a name mismatch between local and pipeline runs, not a logic error in the Terraform code itself.
