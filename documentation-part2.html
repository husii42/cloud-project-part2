1. What is this?

Part II turns the infrastructure from Part I into a working application: a small Flask web app that stores and displays images in the Storage Account, plus a CI/CD pipeline that deploys both the infrastructure and the application automatically.

2. The Application

The application is a small Flask app (application/app.py) with two routes, matching the two required web pages:

Web Page 1 (/) lists every blob currently in the images container — name, size, last-modified date — with a direct download link for each, and a link to Web Page 2.

Web Page 2 (/upload) is a simple HTML form that lets a user choose a file and upload it. Filenames are sanitized (secure_filename) and prefixed with a UTC timestamp so two uploads with the same original filename never overwrite each other.

Both routes talk to Azure Blob Storage through the azure-storage-blob SDK, authenticated via DefaultAzureCredential — never a connection string or access key. Locally, that credential chain falls back to the developer's az login session; on App Service, it automatically picks up the Web App's System-Assigned Managed Identity.

3. Authentication & Identity Context

Part I already introduced a Managed Identity for the App Service, in anticipation of Part II. Part II completes the picture by switching Key Vault from (legacy) Access Policies to RBAC role assignments, and by granting the App Service's identity access to both Storage and Key Vault the same way:

The two App Service role assignments are defined in the root module rather than inside modules/storage or modules/keyvault. The reason is a subtle Terraform pitfall: the Web App's Managed Identity principal_id is unknown until the Web App is actually created (known after apply). A count expression that branches on whether that value is empty cannot be evaluated during terraform plan, which causes Terraform's "Invalid count argument" error on the very first apply. Defining the role assignments unconditionally at the root - with no count - avoids the problem entirely; Terraform simply defers them until the identity exists.

The pipeline itself authenticates without any stored secret. Azure DevOps' "App registration (automatic)" Service Connection type creates a Service Principal and a federated credential trust in one step - the pipeline exchanges a short-lived OIDC token for an Azure access token at run time, so there is no client secret to leak, expire, or rotate. This is the modern replacement for a Service Principal with a stored password, and matches the "use of SPN" pattern from the Part I feedback - just without the secret.

4. CI/CD Pipeline

The Azure DevOps pipeline (pipelines/azure-pipelines.yml) has three stages:

1. TerraformPlan   – init, validate, plan; publishes the plan as a pipeline artifact
2. TerraformApply  – downloads the plan, applies it (main branch only, "production" environment)
3. DeployApp       – packages application/ into a zip, deploys it to the App Service

Splitting plan and apply into separate stages means the plan can be reviewed (and, if an "environment" approval check is configured in Azure DevOps, explicitly approved) before anything is actually changed in Azure - the same separation of concerns terraform plan vs. terraform apply already enforces locally.

The deploy stage uses Azure App Service's Oryx build (SCM_DO_BUILD_DURING_DEPLOYMENT=true, set both by Terraform and by application/.deployment), so the zip only needs to contain source files - dependencies from requirements.txt are installed on the server during deployment, not bundled into the artifact.

5. Production Timeout Fix

During testing, the application occasionally returned a 502/504 on its very first request after a cold start (e.g. after a deployment, or after the App Service scaled back up from idle). The root cause: Gunicorn's default worker timeout is 30 seconds, and on a cold start the worker has to do more than just start Flask - it also resolves DefaultAzureCredential, which on App Service means a round-trip to the Instance Metadata Service (IMDS) to fetch a Managed Identity token. Under load or during a cold start, that round-trip occasionally took longer than 30 seconds, and Gunicorn killed the worker mid-request.

The fix is in the App Service's startup command (modules/appservice/main.tf): 

gunicorn --bind=0.0.0.0:8000 --timeout 600 --workers 2 --access-logfile '-' --error-logfile '-' app:app

--timeout 600 gives workers ten minutes before being recycled, comfortably covering the credential handshake and any slow Storage API calls. --workers 2 fits the B1 plan's single vCPU without the two workers starving each other of CPU. The log file flags send Gunicorn's own logs to stdout/stderr so they appear in Azure's Log Stream instead of disappearing.

6. Known Limitations (Part II)

No authentication on uploads

Anyone who can reach /upload can upload a file - there is no login, rate limiting, or file-content validation beyond extension checking. This is acceptable for a university exercise, but would need addressing (e.g. Azure AD authentication on the App Service, antivirus scanning of uploads) before handling untrusted public traffic in a real deployment.

Manual Service Connection setup

Creating the Azure DevOps Service Connection with Workload Identity Federation cannot be done from inside this Terraform configuration without introducing a different secret (a Personal Access Token for the azuredevops Terraform provider). It is therefore a documented, one-time manual step - see documentation/AZURE_DEVOPS_SETUP.md - rather than something terraform apply handles end-to-end.

Single environment, single region

As in Part I, only a dev environment in Sweden Central exists. Promoting a change through e.g. dev → staging → prod, or adding region failover, is out of scope for this project.
