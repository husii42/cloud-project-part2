Architecture Rationale — Infrastructure & Pipeline

This document explains why the infrastructure and CI/CD pipeline are designed the way they are — the reasoning behind each decision, not just the mechanics. Intended to support the project write-up / defense.

1. Module structure: one module per resource type

modules/
├── storage/      → Storage Account + Container
├── keyvault/      → Key Vault + secret + role assignments
└── appservice/    → App Service Plan + Linux Web App + Managed Identity

Why this shape: Each module owns exactly one Azure resource type and its directly-dependent sub-resources (e.g. storage owns the container, not just the account). This mirrors how the resources are actually managed in Azure — you’d reason about “the storage layer” or “the secrets layer” as a unit — and keeps each module’s variables.tf/outputs.tf small enough to review at a glance. The alternative (one flat main.tf for everything) becomes unreadable past a handful of resources; the alternative in the other direction (one module per individual resource, including sub-resources) creates excessive indirection for resources that are never meaningfully used independently of their parent.

Why role assignments for the App Service’s identity live in the root module, not inside storage/keyvault: This was a deliberate correction made after testing surfaced a real Terraform limitation. The App Service’s Managed Identity principal_id is (known after apply) until the Web App actually exists. A count expression that branches on whether that value is set (count = var.principal_id != "" ? 1 : 0) cannot be evaluated during terraform plan, because the value the condition depends on isn’t known yet — this is Terraform’s documented behavior, not a bug, but it produces a hard failure (“Invalid count argument”) on the very first apply. Placing the two role assignments in the root module without a count sidesteps the problem entirely: Terraform simply defers creating them until the identity is known, with no conditional logic needed. This is a clean illustration of how Terraform’s dependency graph, not just the README, should drive where a resource is declared.

2. RBAC instead of Key Vault Access Policies

Why: Access Policies are Key Vault’s legacy, vault-specific authorization model — permissions are stored on the vault object itself and don’t compose with the rest of Azure’s permission model. RBAC role assignments are evaluated through the same Microsoft.Authorization system used everywhere else in Azure (Storage, subscriptions, resource groups), which means:

  • One mental model for “who can do what” across the entire infrastructure, not a special case for Key Vault.
  • Permissions are queryable and auditable the same way for every resource (az role assignment list --scope ...), instead of needing a vault-specific API.
  • It matches how the Storage Account’s access for the App Service’s Managed Identity is already granted (Storage Blob Data Contributor), so the project doesn’t mix two different authorization paradigms for conceptually similar “can this identity touch this resource’s data” questions.

Trade-off worth noting: RBAC role propagation can take a few minutes, versus Access Policies which apply near-instantly. For a course project this is a minor inconvenience; in this project it was also the direct cause of several 403 Forbidden errors during initial pipeline setup — not because RBAC was the wrong choice, but because role assignments must be created (and given time to propagate) before the identity that needs them is first used, which is easy to overlook when standing up a pipeline for the first time.

3. Why the application never holds a credential

The Storage Account’s connection string is still written to Key Vault as a secret (storage-connection-string) — this is kept from Part I, mainly to demonstrate the original “store sensitive data in Key Vault” requirement and for manual debugging. The running Flask application does not read it. Instead, app.py uses DefaultAzureCredential, which:

  • Resolves to the App Service’s System-Assigned Managed Identity when running in Azure — Azure handles credential issuance and rotation entirely outside the application; there is no secret to leak, expire, or accidentally commit to Git.
  • Falls back to the developer’s az login session when running locally — the same code path works in both environments without an if environment == "production" branch.

Why this matters beyond “best practice”: a connection string in an App Setting is visible to anyone with read access to the App Service configuration (which, on a student subscription with a single Contributor-level account, might not seem like a meaningful boundary — but on a real team it is exactly the kind of credential that ends up pasted into a Slack message or a support ticket). Removing the credential from the application’s reach entirely removes that failure mode rather than mitigating it.

4. Why the pipeline authenticates via Workload Identity Federation, not a stored secret

Classic Service Principal authentication requires a client secret (or certificate) stored as a pipeline variable/secret — something that exists, needs rotation, and is a credential an attacker could exfiltrate from pipeline logs, variable groups, or a compromised agent. Workload Identity Federation (OIDC) instead has Azure DevOps present a short-lived, signed token to Microsoft Entra ID at run time, which Entra ID exchanges for an Azure access token after verifying the trust relationship — no long-lived secret exists anywhere, in Azure DevOps or in Azure.

This also explains why the Service Connection is set up manually through the Azure DevOps UI (“App registration (automatic)”) rather than via Terraform: creating the federated credential trust through azuredevops/azuread Terraform providers would require a Personal Access Token to authenticate Terraform to Azure DevOps itself — reintroducing exactly the kind of long-lived secret this design is trying to eliminate. The UI-driven, one-time manual step is the smaller attack surface, even though it’s the less “everything is code” option.

5. Why the pipeline needs more than Contributor

A natural first instinct is “the pipeline manages infrastructure, so give it Contributor.” This undershoots what the Terraform configuration actually does:

  • Contributor covers creating/modifying/deleting Azure resources (storage accounts, web apps, vaults) — the management plane.
  • It does not cover creating or deleting RBAC role assignments (Microsoft.Authorization/roleAssignments/*) — Azure deliberately separates “can manage resources” from “can grant access to resources”, since the latter is a privilege-escalation-adjacent capability. This needs User Access Administrator.
  • It also does not cover Key Vault secret data-plane operations (getSecret, setSecret) — RBAC for Key Vault separates “manage the vault resource” from “read/write what’s inside it.” This needs Key Vault Secrets Officer scoped to the vault.

This three-role split is not incidental complexity — it’s Azure’s RBAC model working as designed, and it directly reflects the separation of concerns in the Terraform configuration itself: managing resources, granting access to resources, and reading/writing secrets are three distinct operations, each gated by its own permission.

6. Why Terraform state is remote, with a separate bootstrap step

Local state (terraform.tfstate on a single developer’s machine) breaks the moment more than one actor needs to run terraform apply against the same infrastructure — and a CI/CD pipeline is, by definition, a second actor. Remote state in a Storage Account gives the pipeline and any local developer a single, shared source of truth, with locking to prevent two concurrent applies from corrupting each other.

Why bootstrap is a separate, manually-run configuration instead of part of the main one: Terraform’s state for the main configuration would need to live in a Storage Account that the main configuration itself creates — a circular dependency (the resource doesn’t exist yet when Terraform needs somewhere to record that it’s creating it). Splitting this into a one-time, separately-initialized bootstrap/ configuration (which uses its own short-lived local state, just for this one step) is the standard resolution: it exists specifically to solve a problem that exists before the main configuration can run at all.

7. Why the pipeline splits Plan and Apply into separate stages

TerraformPlan → TerraformApply → DeployApp

Separating plan from apply (rather than running both inline in one job) preserves the same review opportunity that running them separately on the command line gives a developer: the plan’s output can be inspected — and, if an Azure DevOps Environment approval check is configured — explicitly approved, before anything is actually changed in Azure. Passing the plan as a build artifact between stages also guarantees apply acts on exactly the reviewed plan, not a plan freshly regenerated against whatever the infrastructure happens to look like by the time apply runs (which could differ if state changed in between, however briefly).

Why deployment jobs, not plain job blocks: deployment jobs let the production Environment be referenced, which is what enables the optional approval-gate behavior above — a plain job has no equivalent concept. The trade-off (deployment jobs don’t auto-checkout the repository, unlike plain jobs) is a real one we hit directly: it produced an “initialized in an empty directory” Terraform error until checkout: self was added explicitly as the first step. That’s a one-line fix once understood, and a small price for the approval-gate capability gained.

8. Why the App Service’s startup command sets explicit Gunicorn flags

gunicorn --bind=0.0.0.0:8000 --timeout 600 --workers 2 --access-logfile '-' --error-logfile '-' app:app
  • --timeout 600: Gunicorn’s default 30-second worker timeout is tuned for steady-state request handling, not cold starts. On App Service, a cold-started worker also has to complete the DefaultAzureCredential → Managed Identity token handshake (a round trip to Azure’s Instance Metadata Service) before it can serve its first request — under load or right after a deployment, that combination occasionally exceeded 30 seconds, causing Gunicorn to kill the worker mid-request. 600 seconds gives ample headroom without masking a genuinely broken request (which would still eventually need investigating, just not be falsely attributed to a timeout).
  • --workers 2: matched to the B1 App Service Plan’s single vCPU — enough to handle one request while another is in flight, without over-provisioning workers that would compete for the same limited CPU.
  • --access-logfile '-' --error-logfile '-': sends Gunicorn’s own logs to stdout/stderr, which is what Azure’s Log Stream (az webapp log tail) actually surfaces — without this, Gunicorn’s logs would go nowhere visible, making any future production issue much harder to diagnose.

9. Why project_name has no random suffix

Earlier iterations of this project (and the grading feedback that preceded Part II) used a random_string suffix to guarantee globally-unique Storage Account/Key Vault names automatically. Part II deliberately removes this, per explicit project direction, in favor of the developer choosing a unique name themselves via terraform.tfvars (and the equivalent PROJECT_NAME pipeline variable).

Trade-off accepted knowingly: this shifts the uniqueness burden onto the person running the project, and — as this project’s own setup process demonstrated at length — makes the project more sensitive to a single inconsistency (local terraform.tfvars vs. the pipeline’s variable falling back to a different default) than a random suffix would have been. The mitigation isn’t a code-level safeguard but a process one: pick a name once, write it down, and set it identically everywhere a project_name value is read from. The SETUP_GUIDE.md documents this as the single most important step to get right before starting.
